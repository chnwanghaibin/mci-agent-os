## 1. 一句话定义

一个面向**非技术业务用户**的 Agent 训练工作台：用户通过提供案例和判断对错，把 Agent 在某类任务上**越练越精、且不退步**；系统负责把判断转化为可解释的配置改动，全程对用户白盒可见。

---

## 2. 背景与问题

### 2.1 现有产品的两个停滞态

现有同类产品普遍停在：

- **配置态**：用户能调 prompt、绑知识库，但改完不知道变好还是变坏，改进靠玄学。
- **成绩单态**：评估模块只展示分数，没有从失败回流到配置的闭环。

真正缺失的是**训练闭环**：评估失败 → 自动诊断 → 系统提出可执行改动 → 人工审核落地。

### 2.2 被低估的两个约束

**用户约束**：目标用户是合规专员、运营、客服主管等不懂代码的业务人员。他们不会写 prompt，但极擅长判断"这个回答对不对"。产品必须把工程复杂度全部隐藏，只暴露"判断"这一个动作。

**数据约束**：业务用户手上通常没有标好的评估案例，且案例数量少（几十条量级，不是几千条）。系统必须在小数据量下仍能给出有意义的训练信号，并诚实告知什么情况下结论不可靠。

---

## 3. 产品目标与设计原则

**核心理念：用户当裁判，LLM 当工程师。**
用户只提供"什么是好的标准"和"这条对不对"，所有 prompt、规则、示范、检索配置的改动由系统在背后生成，用户从头到尾不需要看到"prompt"这个词。

**四条设计原则：**

- **P1 评估驱动**：评估不是事后考试，而是产品的中心动作；Agent 改进是评估循环掉出来的副产品。
- **P2 全程白盒**：Agent 可训练状态拆成一组具名、可读、可单独编辑的单元，绝不优化成一坨不可读的 prompt。用户随时能看到"改了什么、为什么改"，随时能接管。
- **P3 声明式配置**：用户写的一切都是声明式 config，不直接执行；运行时把它"编译"成一次具体的上下文与执行图。由此天然得到版本化、diff、回滚、A/B、草稿/发布分离。
- **P4 训练 = 配置优化，不是调权重**：底层是"评估失败 → 反推 config 改动 → 守住回归"的闭环。对用户话术可叫"训练"，底层是可解释、即时生效、可回滚的配置演进。

**新增原则：**

- **P5 诚实对待小数据**：当案例数量不足以支撑统计可信的评估结论时，系统主动告知，不用不可靠的数字误导用户。
- **P6 渐进式信任**：功能不强制用户一次性完成所有配置，每一步都有最小可用路径，复杂能力随数据积累逐步解锁。

## 4. 用户与角色

| 角色 | 典型身份 | 在平台里做什么 |
|---|---|---|
| **业务训练者** | 合规专员、运营、客服主管 | 创建 Agent、提供案例、当裁判、审查改动、发布（v1 主角） |
| 资产维护者 | 业务骨干 / 小管理员 | 维护统一知识库、规则库、工具库等共享资产 |
| 平台管理员 | IT / 平台方 | 配额治理、权限、模型路由、接口鉴权 |
| 接入方（外部） | 下游业务系统 | 通过接口中心调用已发布 Agent |

## 5. 核心概念与领域模型

| 实体 | 说明 | 关键关系 |
|---|---|---|
| Agent | 专精某类任务的智能体，有用途、状态、当前版本指针 | 属于 User；指向一个当前 AgentVersion |
| **AgentVersion** | Agent 可训练状态的**不可变快照**，带质量分和创建来源（训练/手动编辑） | 由"训练轮"或"人工编辑"产生，可 diff、可回滚 |
| 白盒配置单元 | 指令分段 + 规则绑定 + Few-shot 示范 + 检索配置 + 参数 | 是 AgentVersion 的内容，每项均可单独 diff |
| KnowledgeDoc / Chunk | 用户级统一知识库 | Agent ⇄ Doc 为 M:N，**绑定关系上挂 per-agent 检索配置** |
| Rule | 用户级统一规则库的卡片 | 分为**共享规则**（影响所有引用 Agent）和 **Agent 私有草稿规则** |
| **Rubric / Criterion** | Agent 的评分标准，结构化多维度 | 一等实体，被 Judge 复用、被版本追踪 |
| **EvalCase** | 评估用例（输入/期望/用户判断/来源/切分标记 train\|holdout） | 来源：用户手动创建、系统辅助生成、生产调用回灌 |
| EvalRun | 某 AgentVersion 在某评估集上的一次跑分 | 附带 Rubric 版本快照，确保历史可比 |
| **FailureCluster** | 一批失败案例聚合成的失败主题，附自然语言根因 | 是 Proposer 的直接输入，而非原始失败案例 |
| **Proposal** | 一轮训练中，系统对某个白盒单元提的一条改动 diff | 带诊断理由、触发案例链接、依赖关系标注、审查状态 |
| TrainingRound | 一次训练：评估 → 聚类诊断 → 提案 → 审查 → 守门 → 升版 | 串联 EvalRun / FailureCluster / Proposal / 新 AgentVersion |

**核心设计约束：知识/规则/工具一律"引用"而非"拷贝"**。改一处，所有引用它的 Agent 同步生效；真正 per-agent 的是绑定关系上的配置，不是资产本身。但训练过程中产生的**新规则**，必须先以私有草稿形式存在，经资产维护者审批后才能升级为共享规则。

## 6. 功能模块详述

### 6.1 Agent 管理

**生命周期状态：** 草稿 → 待训练 → 已训练 → 已发布

每个 Agent 有用途描述、类型、当前质量分、状态、当前激活版本。支持新建、复制（继承资产绑定与当前版本作为新起点）、启停、删除。

Agent 详情页是训练闭环的核心入口，分为以下区域：
- **配置概览**：白盒单元当前状态一览
- **案例管理**：评估案例列表 + 冷启动引导
- **训练历史**：历次训练的 Timeline，可点击查看每次训练的 Proposal 决策
- **版本列表**：所有 AgentVersion，支持 diff 两个版本、一键回滚
- **实时评估**：随时触发评估，查看当前分数

**版本追踪原则（v2.0 新增）：** 所有配置变更——无论来自训练还是手动编辑——都必须产生新的 AgentVersion 快照，保证完整可追溯性。手动编辑版本在 timeline 里用不同图标区分，标注创建者。

---

### 6.2 统一知识库

用户级文档库，支持上传、分类、打标签、片段预览、增删。

**分类采用 metadata tag + filter**，支持同一文档归多类，不用刚性目录。

**检索测试面板**：输入查询词，即时看命中的文档片段和相似度分数，是用户调检索效果的直接反馈面。结果展示里明确标注：哪些 Agent 绑定了这份文档、命中时使用的是哪个 Agent 的检索配置。

**文档更新通知（v2.0 新增）**：当知识库文档更新后，系统检测到所有引用该文档的 Agent，在其详情页展示提示："关联文档《XX》已更新，建议重新评估以确认 Agent 表现未受影响。"这是被动提示，不强制触发重评估。

---

### 6.3 统一规则库（含隔离设计）

**规则分为两类：**

| 类型 | 说明 | 谁能修改 | 影响范围 |
|---|---|---|---|
| 共享规则 | 用户级规则库中的标准规则卡片 | 资产维护者 | 所有引用此规则的 Agent |
| 私有草稿规则 | 训练产生的、尚未提升为共享规则的 Agent 专属规则 | 业务训练者 | 仅该 Agent |

**训练产生的规则提案（Proposal）行为：**
- `rule-binding` 类型（启用/停用某条现有共享规则）：仅影响该 Agent，安全，直接应用。
- `rule-create` 类型（新建规则）：自动创建为该 Agent 的**私有草稿规则**，不写入共享规则库。若多个 Agent 产生了语义相近的私有规则，系统向资产维护者建议"合并为共享规则"（需显式操作）。
- `rule-content` 类型（修改已有共享规则内容）：**高风险操作**，UI 强制展示受影响 Agent 列表，要求资产维护者级别权限才能确认。业务训练者无法在审查闸门里直接接受此类 Proposal，只能"提交变更申请"。

---

### 6.4 统一工具库

把检索、规则校验、生成、路由、人工复核、接口网关等沉淀为可复用工具。

所有工具统一到一套 I/O schema（输入契约、输出契约、`when_to_use` 描述），内置工具、HTTP/API、MCP server、sub-agent 都套同一壳。Agent 绑定工具时可配置参数覆写，不影响工具本身定义。

---

### 6.5 Agent 白盒配置单元

Agent 的"大脑"不是一坨 prompt，而是以下一组具名单元，每项人都能直接读、直接改：

| 单元 | 说明 | 训练杠杆优先级 |
|---|---|---|
| **Few-shot 示范** | 一组"输入 → 期望输出"的例子 | **最高**（最白盒、最安全、用户最自然） |
| **规则绑定** | 引用规则库中的卡片，可启停 | 高 |
| **检索配置** | 绑哪些知识库、top_k、标签过滤 | 高 |
| **指令分段** | 角色 / 任务 / 约束 / 输出格式，改哪段 diff 哪段 | 中（需人工审慎）|
| **参数** | temperature 等，有界范围，优先由系统自动建议 | 低（谨慎对用户暴露）|

**"自动训练"和"人工编辑"共用同一个编辑面。** 系统训练 = 往这堆单元上提 diff；人工调整 = 人往同一堆单元上提 diff。两者走同一套 审查 → 版本 → 评估 管道。

**参数单元补充说明**：temperature 等参数不直接暴露给业务训练者，而是以"回答风格"（保守 / 均衡 / 发散）等业务语言表达，系统在背后映射到具体参数值。Proposal 若涉及参数修改，系统自动校验变更后的值是否在安全区间内，超出则标红拒绝。

### 6.6 Rubric 建立与校准（三层递进）

Rubric 是整个评估体系的地基，质量直接决定训练信号可信度。采用三层递进策略，每层都有独立价值，不强制用户走到第三层。

#### 第一层：从 Agent 描述生成草稿（零额外负担）

用户在创建 Agent 时已经写了用途描述和验收标准，系统据此自动生成 Rubric 草稿：

**输入（用户填写）：**
- Agent 用途描述（大白话，如"审查门店现金流合约里的收益分配和披露条款"）
- 验收标准（口语化，如"必须识别出口径不一致的问题，且给出具体修改建议"）

**输出（系统生成草稿）：**
- 3-5 个评分维度（附解释：为什么选这几个维度）
- 各维度权重建议（附权重分配逻辑说明）
- 每个维度的打分指引草稿（含"满分示例"和"零分示例"）

用户可对草稿进行增删改，也可完全推翻重写。**草稿只是起点，不是约束。**

#### 第二层：人工判断驱动的主动精化（低负担迭代）

在用户累积了一定量的案例判断后（建议 ≥ 15 条），系统可以主动识别 Rubric 里的模糊地带并触发优化建议：

- 系统发现：用户标记为"失败"的案例里，有 5 条 LLM Judge 打的是"通过"——说明 Rubric 的某个维度定义不够清晰
- 系统把这 5 条案例展示给用户，问："这几条你认为失败，LLM 认为通过。差异在哪个维度？"
- 用户补充说明，系统自动更新那个维度的打分指引

**这一层的 Rubric 迭代对用户来说是可选的**，不影响训练的正常运行，只影响 Judge 的打分准确度。

#### 第三层：Judge 校准（分级策略）

校准的目的是量化 LLM Judge 与人工判断的一致程度。**v1 不设硬门槛**，采用渐进策略：

**阶段 A（前 20 条案例）：纯人工模式**
- 用户的每条 👍/👎 判断直接作为评估分数，不引入 LLM Judge
- 好处：完全不依赖 Rubric 质量，用户能立即开始训练
- 坏处：无法自动评估大量案例，训练规模受限

**阶段 B（20-50 条案例）：人工 + LLM 并行**
- LLM Judge 开始打分，与人工判断并行
- 系统计算两者的一致率，但只展示一致率，不阻断训练
- 当一致率 < 0.6 时，展示分歧最大的案例，引导用户修订 Rubric

**阶段 C（50+ 条案例）：LLM Judge 主导，人工抽样复核**
- LLM Judge 全量自动打分
- 每批次随机抽 8% 发给用户人工复核（用于持续监控一致率漂移）
- 一致率持续 < 0.6 触发告警，暂停 LLM Judge 自动打分，回退到阶段 B

### 6.7 案例管理与冷启动

#### 案例输入结构

案例不是一个"大文本框"，而是引导式结构化表单：

| 字段 | 说明 | 是否必填 |
|---|---|---|
| 案例名称 | 一句话说明这条案例测什么（如"日度采集 vs 月度披露口径冲突"）| 必填 |
| 场景描述 | 这段输入的背景（合约类型、涉及条款等）| 选填 |
| 输入内容 | 实际输入 Agent 的文本 | 必填 |
| 期望结论 | 通过 / 拒绝 / 需人工复核 三选一 | 必填 |
| 期望理由 | 一两句说明为什么是这个结论（作为训练信号的核心） | 必填 |
| 关键考察维度 | 对应 Rubric 的哪个维度（多选）| 选填，但建议填 |
| 数据集划分 | 训练集 / 保留集（默认由系统自动分配）| 选填 |

#### 实时质量检查

提交案例时，系统自动检查以下问题并给出提示（软警告，不阻断提交）：

- **完整性**：输入内容过短（< 50 字）、期望理由过短（< 10 字）
- **一致性**：输入内容看起来与 Agent 任务无关（语义相似度 < 阈值）
- **重复性**：与已有案例的输入相似度 > 0.85，提示"这条与案例 #X 非常相似，建议换角度"
- **有效性**：当前 Agent 版本对此输入已经能给出正确答案（输出与期望结论一致），提示"这条对当前 Agent 是通过的，考虑添加更有挑战性的案例"

#### 覆盖率仪表盘

案例列表页顶部展示当前案例集的"质量概况"：
- 每个 Rubric 维度被覆盖的案例数（热力图）
- 通过 / 失败比例
- 案例多样性评分（输入语义分散程度）
- 当前数据集是否达到"可靠训练"的最小样本阈值

#### 冷启动流程

**Step 1：用户提交第一条锚点案例**

这是整个平台的入口要求——用户必须提交至少 1 条真实案例才能继续。这条案例承担：
- 提供输入文本的格式模板（让系统知道真实输入是什么样的）
- 提供期望结论的表达风格
- 作为生成更多案例的"参照物"

**Step 2：系统辅助生成候选案例**

基于：① 用户的锚点案例、② 绑定的知识库内容（随机采样典型条款）、③ Rubric 维度覆盖需求，系统生成 3-5 条候选案例供用户确认。

展示形式是"卡片流"，每次展示 3 条，每条标注：
- 这条案例针对 Rubric 的哪个维度
- 系统对期望结论的判断（附置信度）
- 用户操作：加入 / 跳过 / 加入并修改

**Step 3：最小可用阈值验证**

当案例总数达到 8 条（建议 5 train + 3 holdout）时，"开始训练"按钮解锁，但附带说明：

> "当前案例较少（3 条保留集），统计结果可能不稳定。建议积累到 30+ 条后训练结论更可靠。"

低于 8 条时"开始训练"按钮灰化，说明原因和差距。

**Step 4：生产调用回灌（v1.1）**

Agent 发布后，每次真实 API 调用都可以作为候选案例。系统按策略（失败率高的、新类型的）推荐给用户判断，判断后自动加入案例集。这是长期最稳定的案例补充来源。

### 6.8 训练闭环（七步）

#### 步骤概览

```
① 数据准备       →   ② 基线评估      →   ③ 失败聚类诊断
     ↓                    ↓                      ↓
④ 生成提案 diff  →   ⑤ 人工审查闸门  →   ⑥ 回归守门
                                                  ↓
                                          ⑦ 升版提交 + 沉淀
```

括号标注对业务用户的可见性：⚙️ 内部 / 🟢 用户操作 / 📊 结果可见

---

#### ① 数据准备（⚙️ 内部）

**前置检查：**
- 最小样本量验证：train ≥ 5 条、holdout ≥ 3 条。不满足时训练终止，给出明确提示。
- 若案例未划分 train/holdout，自动按 80/20 分层抽样划分并写回（此后固定，不随机变化）。
- 保留集一旦划定，在该 Agent 的整个生命周期内不参与任何训练，只用于回归验证。

**样本量与可靠性说明（展示给用户）：**

| holdout 案例数 | 可靠性说明 |
|---|---|
| < 5 条 | ⚠️ 过少，门控结论不可靠，仅供参考 |
| 5-15 条 | ⚡ 一般，结论有参考价值但有较大随机性 |
| 15-30 条 | ✅ 良好，结论基本可信 |
| 30+ 条 | ✅ 充分，结论统计可信 |

---

#### ② 基线评估（⚙️ 内部 / 📊 结果可见）

在 train + holdout 集合上运行当前 AgentVersion，使用 Judge 按 Rubric 打分，记录：
- 各 Rubric 维度的基线分数
- 通过率 / 失败案例列表
- 每条失败案例的 Judge 评语（用于后续聚类）

**附带 Rubric 版本快照**：每次 EvalRun 时将当前 Rubric 的完整快照附加到评估记录，确保历史评分结果之间可比（Rubric 改变后历史分数有不可比性告警）。

---

#### ③ 失败聚类诊断（⚙️ 内部 / 📊 模式摘要可见）

把失败案例按语义聚类，再由 Critic 模型给出每个簇的自然语言根因。

**用户看到的是模式摘要，不是原始失败列表：**

```
发现 3 类失败模式：
┌──────────────────────────────────────────────────────┐
│ 模式 A（7 条）                                        │
│ 数据授权范围 ↔ 披露口径未交叉校验                      │
│ Agent 识别了披露频率，但未与授权采集频率做对比           │
│ 指向单元：Few-shot 缺乏此类示范 + 规则库未覆盖该检查    │
├──────────────────────────────────────────────────────┤
│ 模式 B（3 条）                                        │
│ 风险等级判断偏低（实为高风险，输出中风险）               │
│ 指向单元：指令约束段落未明确高风险判定阈值              │
├──────────────────────────────────────────────────────┤
│ 模式 C（2 条）                                        │
│ 输出格式不完整（缺少修订建议字段）                      │
│ 指向单元：输出格式段落定义不够完整                      │
└──────────────────────────────────────────────────────┘
```

用户可以在此步骤"驳回"某个失败模式的诊断（"这个诊断是错的，真正原因是..."），被驳回的诊断进入 Proposer 时附带用户的纠正意见。

---

#### ④ 生成改动提案（⚙️ 内部）

Proposer 模型基于每个失败聚类（而非原始失败案例）生成针对白盒单元的改动 diff。

**每条 Proposal 包含：**
- 改动的目标单元（类型 + 标签）
- before / after 内容对比
- 自然语言诊断理由（引用具体失败模式）
- 触发案例链接（可点击查看原始案例）
- 系统风险评分（0-1，标注是否高风险）
- 与其他 Proposal 的依赖关系（"与 Proposal B 互补，建议同时接受"）

**提案优先级排序**（训练杠杆优先级从高到低）：
1. Few-shot 示范（最安全，最白盒）
2. 规则绑定（启用/停用）
3. 检索配置
4. 指令分段
5. 参数（谨慎）

**系统在提案生成阶段自动拦截：**
- 重复已知无效提案（历史被拒绝的相同方向）
- 涉及修改共享规则内容的提案（降级为"申请变更"，不进入普通审查流程）
- 风险分 > 0.8 的提案（标红警示，不自动拒绝，让用户决定）

---

#### ⑤ 人工审查闸门（🟢 用户操作 — 闭环里唯一的用户动手处）

以卡片形式逐条呈现提案，每张卡片包含完整上下文，用户决策：**接受 / 编辑后接受 / 拒绝**。

**卡片设计要点：**
- diff 视图：红色背景显示 before，绿色背景显示 after
- 诊断理由用业务语言表述，不出现技术术语
- 触发案例可展开查看原文
- 高风险提案用橙色边框 + ⚠️ 标注
- 依赖提案用链接标注（"与卡片 #2 互补"）
- 用户选择"编辑"时，打开内联编辑框，修改 after 内容，保存后状态变为"已编辑"

**全部提案审查完成后**，展示汇总：
- 已接受 N 条 / 已编辑 N 条 / 已拒绝 N 条
- "提交并进入回归守门"按钮

用户此时还可以重新打开某张卡片修改决策。

---

#### ⑥ 回归守门（⚙️ 内部 / 📊 结果可见）

将用户接受/编辑的提案应用为候选版本，在 train + holdout 集合上重新评估。

**守门标准：**
- 训练集分数必须提升（基于样本量动态调整阈值，见技术文档）
- 保留集分数不得下降超过容忍范围

**守门失败时的自动处理：**

系统按提案的风险评分，贪心地逐一剔除高风险提案再重测。找到最小剔除集合使门控通过后，自动拒绝被剔除的提案，说明原因：

> "⚠️ 提案 #3（规则新增：数据授权范围规定）导致保留集分数下降，已自动拒绝。其余 3 条提案顺利通过守门。"

用户看到守门结果，可选择"接受剩余提案并升版"或"撤销本次训练"。

---

#### ⑦ 升版提交与沉淀（⚙️ / 📊 结果可见）

守门通过后，生成新的不可变 AgentVersion 快照，版本号递增（v1.2 → v1.3）。

**沉淀机制：**
- 通过审查 + 守门的 Few-shot 提案，写入该 Agent 的 Procedural Memory（带置信度标注，v1 阶段不跨 Agent 共享，避免污染）
- 原来失败、现在通过的案例，标记为"已修复"，进入回归集——日后只要此案例退化，立即被抓出
- Timeline 自动记录本次训练的摘要（改了什么、分数变化、门控结果）

### 6.9 评估追踪与可观测

#### Step 级执行 Trace

**每次 Agent 执行留完整的步骤级 trace**，是用户 debug 的唯一抓手。

Trace 包含的步骤级信息：

| 步骤 | 记录内容 |
|---|---|
| 知识检索 | 查询词、命中文档列表（含相似度分）、未命中的相关文档（若有）|
| 规则校验 | 应用的规则列表、每条规则是否命中、命中理由 |
| LLM 推理 | 完整 prompt（可折叠）、输出、token 数、耗时、成本 |
| 工具调用 | 工具名称、输入参数、输出结果 |

**Trace 的使用场景：**
- 用户判断某条案例"失败"时，可以点开 trace 看"Agent 为什么给出这个答案"
- Proposer 在生成提案时，trace 数据作为诊断依据（"检索阶段未命中目标文档 → 提案修改检索配置"）
- 生产 API 调用出问题时，支持用 run_id 查询对应 trace 进行 debug

#### 评估追踪仪表盘

- 当前版本质量分（分 Rubric 维度展示）
- 历史分数趋势（折线图，每个版本一个点，hover 可查看该版本的 Proposal 决策）
- 失败案例列表（可筛选：最近失败 / 长期失败 / 新出现失败）
- 版本对比：选择任意两个版本，展示 diff 和评分对比

### 6.10 接口中心与发布

发布 = 将某个 AgentVersion 冻结为对外服务版本（Stable 版本，只读）。

- 草稿 / 已训练 / 已发布 三态分离，发布不影响训练进行中的版本
- 已发布版本提供带鉴权的 API，附带调用调试器和调用记录
- 支持 A/B 分流（v1.1）：同时发布两个版本，按流量比例分配请求，对比效果
- 发布操作不可撤销（只能发布新版本替代），但旧版本仍然保留在版本历史中

### 6.11 工作流稳定性：SOP + 输出锚点 + Skill 库

#### 背景问题

工作流步骤只有自由文本描述时，Agent 在不同运行中对同一步骤的输出格式差异很大：有时返回完整结构，有时只返回摘要，有时遗漏关键字段。这使得下游步骤、人工审核和评估 Judge 无法可靠地解析 Agent 的中间产物。

**根本原因**：自由文本描述约束的是推理方向，但不约束输出形状。

#### 核心方案：Output Anchor（输出锚点）

每个工作流步骤除了自由文本描述（SOP 说明）外，可以附加一个 **输出锚点（Output Anchor）**，定义该步骤 Agent **必须**产出的结构化字段：

| 字段属性 | 说明 |
|---|---|
| `key` | 字段名（如 `conclusion`、`risk_level`） |
| `type` | 字段类型：`enum` / `bool` / `text` / `number` / `list` |
| `options` | 当 type=enum 时，列出所有合法选项 |
| `required` | 是否必须产出（true/false） |
| `description` | 字段含义说明，同时作为 prompt 注入内容 |
| `constraint` | 跨字段约束（如"若 risk_level=高 则 requires_human_review 必须为 true"） |

**Strict 模式**：开启时，步骤的输出中不允许出现锚点以外的顶层字段（用于高度标准化的场景，如合规审查）。

**好处**：
- 自由文本描述不受约束，训练师依然可以细粒度编辑 SOP 描述
- 输出锚点独立于 SOP 描述管理，可单独版本化和评估
- Agent 推理过程（Chain of Thought）不受限制，只有最终产出受约束
- 后续步骤可以可靠地引用前置步骤的字段（`{{step1.risk_level}}`）

#### Skill 库

**Skill = 一组工作流步骤 + 对应的输出锚点定义**，打包成带版本号的可复用单元。

| 属性 | 说明 |
|---|---|
| `name` | Skill 名称（如"合规性预审"、"风险等级评估"） |
| `version` | 语义版本号（如 v1.2） |
| `category` | 分类（合规 / 风控 / 运维 / 通用等） |
| `steps` | 步骤列表，每步含 title + description + OutputAnchor |
| `linkedAgentCount` | 当前引用该 Skill 的 Agent 数量 |

**复用机制**：
1. 在 Agent 的工作流编辑器中，点击"从 Skill 库导入…"
2. 选择 Skill，其所有步骤（含锚点定义）一键导入为该 Agent 的工作流步骤
3. 导入的步骤携带 `skillRef` 标记，记录来源 Skill ID
4. 导入后步骤可本地编辑，不影响 Skill 库中的原始定义
5. 如需升级到新版 Skill，重新导入并手动 diff

**生命周期管理**：
- 初始由平台管理员或资产维护者创建（对应规则库的"资产维护者"角色）
- 业务训练师可查看 Skill 详情、引用 Skill，但不可直接修改 Skill 库（需提变更申请）
- 每次修改需递增版本号，旧版 Skill 引用不自动升级（防止意外破坏已稳定的 Agent）

#### 输出锚点在训练闭环中的作用

| 训练环节 | 锚点作用 |
|---|---|
| **自动 Judge** | Judge 可直接对比锚点字段与期望值（无需解析自由文本） |
| **失败聚类** | 按"哪个锚点字段不符合期望"聚类，精准定位改进点 |
| **Proposal 生成** | 系统可针对特定字段类型（如 enum 字段的错误选项）生成定向改进建议 |
| **版本 diff** | 字段级 diff，而非文本级 diff，可读性更高 |

#### 前端呈现

**Skill 库页面**（独立导航入口）：
- 展示所有 Skill 卡片：名称、版本、分类、步骤数、输出字段总数、已关联 Agent 数
- 可按分类过滤
- 展开每张 Skill 卡，查看步骤列表 + 每步的锚点字段（类型彩色标签展示）
- 约束条件（跨字段规则）单独列出
- 支持新建 Skill

**Agent 工作流编辑器（集成锚点）**：
- 每个工作流步骤卡片右上角显示"输出锚点 (N)"按钮，点击展开锚点编辑器
- 锚点编辑器支持：添加/删除字段、设置类型和选项、标记 required、填写 description 和 constraint
- Strict 模式复选框
- 步骤卡片顶部提供"从 Skill 库导入…"下拉，导入后步骤显示 Skill 来源徽标

## 7. 端到端使用案例

**场景**：合规专员小李，要训练一个"合同审查 Agent"，让它越来越准。

**第 1 步 · 创建并描述任务**
小李在 Agent 管理点"新建"，填写用途描述：「审查门店现金流合约里的收益分配、披露义务、提前终止和争议处理条款」。绑上知识库文档，勾选两条规则。没有 prompt 输入框。

**第 2 步 · 系统生成 Rubric 草稿**
系统基于任务描述 + 小李补填的验收标准「必须识别口径不一致、必须给出具体修订建议」，自动生成四维 Rubric 草稿。小李查看后，把"表述清晰"的权重从 20% 调到 10%，并补充了"口径一致性"维度的打分指引说明。5 分钟完成。

**第 3 步 · 冷启动：提交锚点案例**
小李提交第一条案例：一段真实合约片段（日度采集 vs 月度披露口径不一致），期望结论"拒绝"，理由「第5条授权采集频率与第8条披露频率矛盾，属高风险」。系统随即从知识库生成 3 条候选案例，小李确认了其中 2 条，修改了 1 条。重复 3 轮，案例集达到 9 条（6 train + 3 holdout），"开始训练"解锁（附带数据量不足提示）。

**第 4 步 · 首次评估（基线）**
系统跑基线：质量分 72，9 条里失败 3 条。系统把失败聚成一个主题：「数据授权范围 ↔ 披露口径未交叉校验」，并指出根因：「指令约束段落未覆盖此比对，《合规检查清单》有相关条款但检索未命中」。

**第 5 步 · 系统提出 4 条 Proposal**
- ① Few-shot +1：用这条失败合约教 Agent 「日采集 vs 月披露」应判高风险
- ② 检索配置：标签过滤加「数据授权、披露口径」，top_k 4→6
- ③ 规则 +1（私有草稿）：「数据授权范围与披露口径必须匹配」
- ④ 指令约束段落：加一句「必须校验授权采集频率与披露频率是否一致」（风险分 0.7，⚠️ 标注）

**第 6 步 · 人工审查闸门（小李唯一动手处）**
小李接受①；觉得②不够完善，展开编辑，把 top_k 改成 5（不是 6）；接受③（保存为私有草稿规则）；对④，看到 ⚠️ 风险标注，考虑后拒绝（「指令改动太大，先看这三条效果」）。全程业务语言，没碰 prompt。

**第 7 步 · 回归守门**
系统把①②③应用为候选版本，重跑：训练集 72→88（+16%），保留集 70→83（+13%），通过守门。

**第 8 步 · 升版与沉淀**
提交为 v0.2，质量分 72→88。①的 Few-shot 和③的私有规则写入 Procedural Memory。失败的 3 条案例进入回归集。小李确认后，在接口中心发布 v0.2。

**结果**：小李没写一行代码，仅靠「提供锚点案例 + 确认生成案例 + 判断对错 + 审查改动」，把 Agent 从 72 练到 88，且明确知道改了什么。

## 8. 关键风险与对策

| 风险 | 后果 | 对策 |
|---|---|---|
| Judge 未校准 | 在错误标准上越练越偏 | 分级校准策略（6.6 第三层），前 20 条案例纯人工，渐进引入 LLM Judge |
| 小数据集门控失效 | 随机性掩盖真实效果 | 动态阈值 + 样本量提示（6.8 步骤①），不用固定阈值误导用户 |
| 训练天花板 | 用户误以为可无限变好 | 明示：config 级训练到顶后需换模型/改任务分解/补数据 |
| 黑箱化 | 用户不敢用，出事无法追责 | 全程白盒（P2）+ 人工审查闸门 + Step 级 Trace |
| 规则跨 Agent 污染 | 一个 Agent 的训练影响其他 Agent | 三类规则 Proposal 隔离设计（6.3）|
| 冷启动门槛过高 | 新用户在第一步就放弃 | 零额外负担 Rubric 草稿生成 + 1 条锚点案例启动冷启动（6.7）|
| 提案质量差 | 用户在审查闸门里全拒绝，训练无效 | 失败聚类先于 Proposer（6.8 步骤③），Proposer 输入是模式而非个例 |
| Rubric 版本不一致 | 历史分数不可比 | EvalRun 附带 Rubric 版本快照，Rubric 修改后历史分数有不可比性告警 |
| 强主观任务 rubric 不稳 | 标准立不住，闭环跑不动 | 产品识别此类任务并引导用户：「此类任务主观性强，建议先专注于可量化的维度」|

## 10. 设计决策记录

本节记录产品设计过程中的明确决策，避免实现阶段重复讨论。

---

### D1. Rubric 修改后，历史评估分数的处理

**决策**：保留旧分数，不做重算；在评估历史趋势图中，Rubric 发生变更的时间点标注"断裂点"，并附说明"此版本后 Rubric 已修改，与之前分数使用不同标准，直接对比可能有误导"。

**理由**：重算历史分数成本高（需对所有历史案例重新 Judge），且 Rubric 修订通常是"修正偏差"，用新标准重算旧分数反而会引入新的混淆。断裂点标注让用户清楚地知道"这是两套标准，不可直接比较"，比无声地重算更诚实。

**实现要点**：
- 每次 EvalRun 附带当前 Rubric 的内容哈希快照
- 相邻两次 EvalRun 的 Rubric 哈希不同时，趋势图在该节点渲染断裂分割线
- Agent 详情页 Rubric 编辑区顶部展示提示："修改评分标准会使历史分数不可与未来分数直接比较"

---

### D2. 并发训练运行的锁机制

**决策**：同一 Agent 同一时刻只允许一个 TrainingRun 处于活跃状态（queued / running / awaiting_review / gating 均视为活跃）。用户触发第二次训练时，系统拒绝并提示"当前已有训练任务进行中，请等待完成或取消后重试"。

**理由**：两个并发的 TrainingRun 都基于同一 base_version，都可能产生冲突的 Proposal，最后写入版本时会产生"后写覆盖前写"的数据竞争，导致一次训练的结果被无声丢弃。

**实现要点**：
- 训练启动时检查数据库是否有该 Agent 的活跃 TrainingRun，有则拒绝
- UI 层：当训练进行中时，"开始训练"按钮变为"训练进行中…"并禁用
- 提供"取消当前训练"操作（将 TrainingRun 状态置为 failed，释放锁）

---

### D3. 版本回滚的完整语义

**决策**：回滚操作会创建一个新的 AgentVersion 快照（source_type = 'rollback'），内容与目标版本完全一致，版本号递增（不复用目标版本号）。AgentVersion 链保持线性，不产生分支。已积累的测试案例判断（case_judgments）独立于版本，不回滚，全部保留。

**具体语义**：

| 场景 | 行为 |
|---|---|
| 回滚 v1.5 → v1.3 的内容 | 创建 v1.6，内容 = v1.3 的快照，parent = v1.5 |
| Timeline 中的回滚事件 | 显示"v1.6（回滚自 v1.3）by 用户名" |
| 回滚后的案例判断 | 全部保留，可继续用于下一次训练 |
| 回滚后的 EvalRun 历史 | 全部保留，趋势图中可见回滚节点 |

**理由**：保持 AgentVersion 链线性，避免 git 式"分支合并"的复杂性（v1 阶段不做分支）；用新版本号表示回滚，让 Timeline 清晰展示"曾经回退过"这一历史事实；案例判断是用户劳动的成果，不应因版本回滚而丢失。

---

### D4. 参数（temperature）单元的范围管控

**决策**：业务训练者不直接看到参数数值，而是通过"回答风格"选择器操作（保守 / 均衡 / 发散），系统在背后映射到具体值。Proposer 生成的 `parameter` 类型提案，before/after 内容同样使用业务语言描述，不展示原始数值。

**映射表（v1 固定，后续可配置化）**：

| 回答风格 | temperature | 适用场景 |
|---|---|---|
| 保守（克制） | 0.1 | 合规审查、规则校验等高精度任务 |
| 均衡（推荐） | 0.3 | 大多数业务任务 |
| 发散（创意） | 0.7 | 内容生成、方案建议等开放性任务 |

**系统拒绝的情况**：
- Proposer 提案的参数值超出 [0.0, 1.0] 范围
- 用户手动输入（平台管理员界面）超出模型官方支持范围

**实现要点**：参数在 agent_whitebox 的 `parameters` JSONB 字段存储为数值，仅在展示层做业务语言转换；Proposer Prompt 里参数提案的 after 字段格式为 `{"style": "保守", "temperature": 0.1}`，UI 只展示 style 字段。

---

### D5. 知识库文档更新后的通知触发时机

**决策**：触发时机为"文档内容被替换/更新时"（不含纯元数据变更如标签修改）。通知方式为 Agent 详情页展示一个"知识库已更新"提示 badge，不发邮件/消息，不强制触发重评估。

**具体规则**：

| 变更类型 | 是否触发通知 |
|---|---|
| 上传新版本文档（替换内容）| ✅ 触发 |
| 修改文档标签 / 分类 | ❌ 不触发 |
| 修改文档标题 | ❌ 不触发 |
| 删除文档 | ✅ 触发（且需人工确认）|

**通知内容**：Agent 详情页的知识库绑定区域，显示"⚡ 关联文档《XX》已于 3 天前更新，建议重新评估以确认 Agent 表现未受影响。"用户可点击"已确认，忽略"来消除提示。

**理由**：重评估成本较高，且文档更新不一定影响 Agent 的关键判断路径，让用户自主决定是否评估，比自动触发更符合"低干扰"原则。

---

### D6. 生产 API 调用回灌的数据模型

**决策**：v1 不做自动回灌，但在 production_calls 表预留数据结构，v1.1 基于此实现回灌流程。生产调用数据与评估案例解耦存储，通过用户主动"导入为候选案例"的操作连接。

**数据流**：

```
生产 API 调用
    ↓（自动记录到 production_calls 表）
系统按策略筛选候选（失败率高、输入类型新颖）
    ↓（v1.1 实现）
"待确认案例"队列（展示在案例管理页）
    ↓（用户操作）
用户确认期望输出 → 写入 eval_cases 表（source = 'production'）
```

**v1 实现范围**：只需在 Agent API 调用链路里记录 `production_calls` 基础日志（run_id、input 摘要、output 摘要、verdict、timestamp），不做候选案例生成和回灌 UI。

---

### D7. A/B 分流期间的数据隔离

**决策**：每次生产 API 调用记录所使用的 AgentVersion，评估数据严格按版本隔离，不允许跨版本混用训练数据。

**实现要点（v1.1）**：
- API 调用响应 Header 中返回 `X-Agent-Version: v1.3`
- `production_calls` 表有 `agent_version_id` 字段
- 从生产调用生成候选案例时，候选案例附带来源版本标记
- 用某版本的生产数据训练时，只用该版本产生的调用数据（不混用其他版本）
