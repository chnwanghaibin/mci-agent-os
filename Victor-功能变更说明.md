# Victor 功能变更说明

> 本文档记录在 Wang Haibin 原始前端 Demo 基础上，Victor 新增和修改的功能逻辑。
> 不含代码实现，仅描述行为、设计意图和决策。

---

## 原始版本基础能力（Wang Haibin）

Wang Haibin 的原版包含平台基础骨架：

- 多页面导航框架（矩阵看板 / Agent 管理 / 知识库 / 规则库 / 工具库 / 评估 / 接口中心）
- Agent 创建与状态管理（草稿 / 已训练 / 已发布三态）
- 工作流步骤基础编辑（标题、描述、启停开关）
- 知识库、规则库、工具库的基础列表展示
- Rubric 多维评分结构与评估集基础框架
- 训练控制台基础（运行训练、查看 trace 步骤）
- Agent 详情页多面板布局（配置 / 工作流 / 评估 / 训练 / 接口）

---

## 新增与修改功能

### 1. 矩阵看板（Agent 定位系统）

**新增。** 在顶层导航增加矩阵视图，以"业务流程 × 业务节点"二维表格展示整个 Agent 部署全景。

- 每个格子代表一个业务场景位置，Agent 放置于对应位置
- 格子颜色区分状态：已发布（绿）/ 已训练（蓝）/ 草稿（灰）/ 空缺（白）
- 点击格子可跳转至该 Agent 的详情页
- 平台整体覆盖率一目了然，空白格子提示待填补位置

**设计意图**：滴灌通的业务是有结构的（6 条流程 × 多个节点），Agent 不是孤立的，是按业务位置部署的。矩阵视图让团队在"全局部署地图"视角下管理 Agent，而不是在一个平铺列表里找。

---

### 2. Blueprint 蓝图自动生成

**新增。** 新建 Agent 时，用户输入任务描述，系统根据关键词匹配自动识别业务类型（合同审查 / 工单分诊 / 合规检查），并自动填充：

- 推荐工作流步骤（有序 SOP）
- 推荐工具绑定
- 推荐测试用例模板
- 相关业务说明

**设计意图**：降低 Agent 创建的冷启动门槛。非技术用户不需要从零开始配置，蓝图给出一个合理起点，用户在此基础上调整，而非面对空白页面。

---

### 3. Skill 库（可复用 SOP 步骤包）

**新增。** 独立于 Agent 的 Skill 库，每个 Skill 是一组命名的、版本化的工作流步骤。

- 工作流步骤可引用一个 Skill，相当于复用该 Skill 的标准执行路径
- Skill 独立版本化，更新后引用它的 Agent 步骤可主动升级
- 每个 Skill 步骤也有自己的输出锚点（OutputAnchor）

**设计意图**：类似编程里的函数复用。多个 Agent 做同一种操作（如"检索审查标准"）时，不应各自维护一份步骤描述——改一处就要改所有地方。Skill 是对 SOP 知识的沉淀和共享。

---

### 4. OutputAnchor（输出锚点）

**新增。** 每个工作流步骤可定义一组结构化输出字段，作为该步骤的"完成标准"。

- 字段有类型（enum / bool / text / number / list）
- 字段有必填标记
- 步骤有 strict 模式：严格模式下缺字段即失败
- 字段定义是跨字段约束（Constraint DSL）的基础

**设计意图**：工作流步骤如果只有自然语言描述，LLM 的输出是不可验证的。OutputAnchor 把"期望输出是什么"从模糊的说明变成可机器检查的结构，是整个工作流可追溯的基础。

---

### 5. Constraint DSL（跨字段约束）

**新增（本轮）。** 在 OutputAnchor 字段上以点选方式声明跨字段依赖规则，替代原来在 prompt 里写自然语言约束。

**规则形式**：如果 [字段A] [运算符] [值] → 则 [字段B] 必须为 [值]
- 运算符：等于 / 不等于 / 大于 / 大于等于 / 小于 / 小于等于
- 示例："如果 risk_level = 高 → 则 requires_human_review 必须为 true"

**底层运转逻辑（三层）**：

1. **JSON Schema 强制（层 1）**：通过 LLM 的结构化输出接口（tool_use / function calling）在解码层强制字段存在、类型正确、枚举合法。此层不走 prompt，稳定率约 99%。但无法处理跨字段依赖。

2. **Constraint DSL 事后验证 + 处理（层 2）**：LLM 输出后立即由执行引擎评估每条约束规则，并按策略处理违反：
   - `override`（机械覆写）：约束条件成立时直接写值，100% 确定性，trace 中标注"机械覆写"
   - `retry`（带反馈重试）：将违反信息注入 prompt 重新调用，最多 2 次，约 95-99% 稳定
   - `escalate`（升人工）：无法自动修复时标记待人工处理

3. **Trace 记录（层 3）**：每次约束检查结果（触发 / 违反 / 处理方式 / 修改前后值）写入 step 级 trace，精确可查，不掩盖。

**关键认知**：Constraint DSL 本身不能约束 LLM 的生成过程，它是事后验证器。真正的稳定性来自 JSON Schema（层 1）+ 执行引擎处理（层 2）的组合。对于简单布尔/枚举约束用 override，对需要 LLM 自我修正的复杂约束用 retry。

---

### 6. 步骤 InputFields 声明（上游字段读取）

**新增（本轮）。** 每个工作流步骤显式声明它从哪些上游锚点字段读取数据（通过字段名匹配）。

- 执行引擎按声明自动提取上游字段，无需人工指定搬运路径
- trace 中可精确查看每个步骤"从哪里读到了哪个值"
- 是步骤级追溯和 debug 的基础

**设计意图**：在没有 inputFields 声明时，步骤之间的数据流是隐式的——你只能从自然语言描述里猜"这一步用了哪些上一步的结果"。声明后，数据流变成显式的、可检查的、可在 trace 里验证的。

---

### 7. 路由条件 DSL（结构化路由）

**新增（本轮）。** 工作流步骤的路由条件从自由文本输入改为结构化三元组。

- **原来**：一个文本输入框，用户手写"若 risk_level=高 则跳转"
- **现在**：[字段▼] [运算符▼] [值] 三个下拉/输入，组合成结构化条件

**意义**：文本条件是不可解析的——执行引擎不知道条件是否被满足，只能再问 LLM。结构化条件可以被执行引擎直接计算，路由结果可预测，可在 trace 中精确记录"哪条路由被触发"。

---

### 8. Rule 隔离类型

**新增。** 规则库中的规则对 Agent 有三种绑定方式，决定训练产生的规则变更如何流转：

- **binding（绑定）**：启停已有共享规则，不改规则内容，最安全
- **create（新建草稿）**：训练在 Agent 私有草稿空间创建新规则，不写入共享库，需人工审查后再提交共享
- **content（内容变更申请）**：对共享规则提出内容修改申请，等待规则库管理员审批

**设计意图**：训练不能直接改共享规则库——这会影响所有引用该规则的 Agent。隔离类型确保训练产生的规则变更有明确的流转路径，不会静默污染共享资产。

---

### 9. Judge 校准阶段（judgePhase）

**新增。** Judge 自动化程度分三阶段推进，而不是一开始就全自动：

- **人工标注**（前 20 条）：LLM Judge 不参与打分，全人工
- **平行验证**（20-50 条）：人工与 LLM 同时打分，对比一致率，一致率不达标禁止进入下一阶段
- **自动裁判**（50 条以上）：LLM 主导打分，人工抽检

Agent 详情页展示当前所处阶段，用户可手动查看和调整。

**设计意图**：冷启动时 LLM Judge 的标准还没有被用户行为校准，贸然全自动打分相当于在噪声上优化——训练看起来在进步，但分数对应的标准可能已经漂移。强制走校准阶段确保分数是真实质量的代理。

---

### 10. 测试用例 Train/Holdout 切分

**新增。** 测试用例可手动标记归属：训练集（train）或留出集（holdout）。

- 系统默认按分层抽样自动切分（约 80/20）
- 用户可手动将特定边界场景固定为留出集（如"这条用例是核心回归基准，不能参与训练"）
- holdout 不参与优化，只用于防过拟合验证

**设计意图**：自动切分对大多数场景足够，但业务专家知道哪些用例是"最关键的回归防线"。允许手动控制 holdout 集，让业务判断和技术约束对齐。

---

### 11. Proposal 元数据增强

**新增。** 训练生成的每条改动提案（Proposal）携带额外元数据：

- **riskFlag**：标记该改动有过拟合风险（⚠ 标注），帮助用户在审查闸门时谨慎决策
- **dependsOn**：声明该提案依赖哪些其他提案才能生效（互补关系）
- **conflictsWith**：声明该提案与哪些提案相互冲突，不能同时接受

**设计意图**：业务用户在审查改动时没有全局视角——他们逐条看卡片，不知道"接受 A 但拒绝 B 会破坏 A 的效果"。元数据把这些隐式依赖显式化，让审查决策有足够信息。

---

### 12. Training Lesson Summary（训练摘要卡）

**新增（本轮）。** 每轮训练完成后自动生成一张摘要卡，在下一次启动训练前展示。

**内容三部分**：
- **修复成功**：此次训练消除了哪些失败模式（用户可读的自然语言摘要）
- **拦截记录**：有哪些改动被回归守门拦截、未能写入，及拦截原因（帮助用户理解哪些问题还未解决）
- **建议聚焦**：基于当前剩余失败模式，下一轮建议优先处理的方向

**注入下轮训练**：摘要内容（特别是"拦截记录"）在下一轮训练启动时注入 Proposer 上下文，让 Proposer 不重复提出已被证伪的改动。

**设计意图**：没有摘要时，每轮训练都从零开始——系统不知道上一轮尝试过什么、为什么失败。摘要卡把训练经验显式存档，用户端可读（看到"修了什么"），引擎端可用（不重蹈覆辙）。

---

### 13. 白盒配置单元（Whitebox Units）

**新增 / 扩展。** Agent 的"大脑"被拆解为一组具名可编辑单元，人工编辑和系统训练共用同一个编辑面：

- **指令分段**：角色 / 任务 / 约束 / 输出格式四段独立编辑，改哪段 diff 哪段
- **Few-shot 示范**：以输入→输出例子教 Agent，每条有稳定 ID 用于版本 diff
- **检索配置**：绑定知识库、top_k、标签过滤
- **Rubric**：多维度打分标准，每个维度有权重和打分指引
- **I/O Schema**：声明 Agent 的输入和输出字段列表，作为编排层契约

**设计意图**：反黑箱。如果 Agent 的核心是一大段不透明的 prompt，用户改不了、训练改不了、出了问题也找不到是哪里的问题。拆成具名单元后，训练 = 对这些单元提 diff，人工调整 = 人对同一堆单元提 diff，两者走同一条审查路径。

## 设计原则汇总

以上所有修改都围绕两个核心原则展开：

**反黑箱**：每一层都要让用户看到发生了什么。工作流步骤的输入输出是可见的，约束违反是可追踪的，训练改了什么是可审查的，分数是什么标准下打出来的是可校准的。

**渐进确定性**：不追求一步到位的全自动化，而是在每个环节用结构化声明（Anchor / DSL / Schema）把不确定性逐步压缩，在无法压缩的地方保留人工介入点（审查闸门 / Judge 校准 / 约束升级）。
