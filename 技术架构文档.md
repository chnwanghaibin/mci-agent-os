## 1. 整体架构概览

```
┌─────────────────────────────────────────────────────────────────┐
│                        前端（React SPA）                         │
│   AgentDetailPage → TrainingRunModal → ProposalReviewModal       │
└────────────────────────┬────────────────────────────────────────┘
                         │ REST / WebSocket
┌────────────────────────▼────────────────────────────────────────┐
│                    API Gateway（FastAPI）                         │
│  /agents  /training  /proposals  /eval  /versions  /retrieval   │
└──┬──────────┬──────────┬──────────┬──────────┬──────────┬───────┘
   │          │          │          │          │          │
   ▼          ▼          ▼          ▼          ▼          ▼
 Agent      Training   Proposal   Eval      Version    Retrieval
 Service    Engine     Service    Service   Store      Service
   │          │                    │          │
   │     ┌────▼────┐          ┌────▼────┐    │
   │     │Proposer │          │  Judge  │    │
   │     │ /Critic │          │Calibrat.│    │
   │     └─────────┘          └─────────┘    │
   │                                         │
   └──────────────┬──────────────────────────┘
                  ▼
           PostgreSQL + pgvector
           Redis（队列 + 缓存）
           S3（版本快照 + 评估报告）
```

**关键设计原则：**

- **无状态计算层**：TrainingEngine / ProposalService / EvalService 均为无状态微服务，通过 Redis 任务队列调度。
- **不可变版本**：每次训练完成后，AgentVersion 作为不可变快照写入 S3，永不修改。
- **闸门先行**：Proposal 必须经过人工审查闸门 + 回归门控双重通过，才能合并到 Agent 主配置。
- **Judge 与 Proposer 分离**：评分模型（Judge）与生成建议模型（Proposer/Critic）使用不同的 LLM，避免自我强化偏差。

## 3. 训练引擎（Training Engine）

训练引擎是整个平台的核心，负责协调 6 步训练闭环：

```
① 数据准备 → ② 基线评估 → ③ Proposer生成 → ④ Critic过滤
     → ⑤ 人工审查闸门 → ⑥ 回归门控 → ⑦ 版本写入
```

### 3.1 步骤详解

**① 数据准备（Data Preparation）**

```python
class DataPreparer:
    def prepare(self, agent_id: str, run_id: str) -> PreparedDataset:
        cases = db.query(
            "SELECT * FROM test_cases WHERE agent_id = %s", agent_id
        )
        # 按 split 字段划分；若 split 未设置，用 80/20 随机分层抽样
        train_cases = [c for c in cases if c.split == 'train']
        holdout_cases = [c for c in cases if c.split == 'holdout']
        
        if not train_cases:
            train_cases, holdout_cases = stratified_split(cases, ratio=0.8)
            # 写回 split 字段
            db.bulk_update_split(train_cases, holdout_cases)
        
        return PreparedDataset(train=train_cases, holdout=holdout_cases)
```

```python
async def commit_version(agent_id: str, run_id: str, accepted_proposals: list[Proposal]):
    # 读取当前白盒配置
    whitebox = db.get_agent_whitebox(agent_id)
    
    # 应用已接受的提案
    new_whitebox = apply_proposals(whitebox, accepted_proposals)
    
    # 生成新版本号
    old_version = db.get_active_version(agent_id)
    new_tag = bump_version(old_version.version_tag)  # "v1.2" -> "v1.3"
    
    # 写入不可变快照
    snapshot = new_whitebox.to_dict()
    s3.put_object(f"agents/{agent_id}/versions/{new_tag}.json", json.dumps(snapshot))
    
    new_version = db.insert_version(
        agent_id=agent_id,
        version_tag=new_tag,
        snapshot=snapshot,
        parent_version=old_version.id,
        created_by=f"training-run:{run_id}"
    )
    
    # 更新 Agent 当前激活版本
    db.update_agent(agent_id, active_version=new_version.id, status='trained')
    db.update_training_run(run_id, result_version=new_version.id, status='done')
```

### 3.2 任务队列设计

训练任务通过 Redis + Celery 异步调度：

## 4. Proposer / Critic 设计

### 4.1 Proposer 模型

Proposer 负责分析失败案例，生成白盒改进建议（Proposal）。

**输入上下文：**
```
- 失败 TestCase 列表（input, expected, actual_output, judge_score, judge_feedback）
- 当前 Agent 白盒配置（instructionSegments, fewShots, retrievalConfig, rubric）
- Agent 的历史 Proposal 记录（避免重复生成已知无效方案）
```

**Proposer Prompt 模板（few-shot 示例）：**

```
你是一个 Agent 训练专家。分析下方失败案例，针对白盒配置单元提出最多 5 条改进提案。

== 当前白盒配置 ==
[instruction_segments_json]
[few_shots_json]
[retrieval_config_json]

== 失败案例 ==
[failed_cases_with_scores_json]

== Rubric 评分标准 ==
[rubric_json]

== 历史提案（已拒绝，请勿重复）==
[rejected_proposals_json]

请以 JSON 数组格式输出，每条提案包含：
- unit: "few-shot"|"rule"|"retrieval"|"instruction"|"parameter"
- unit_label: 人类可读标签
- before: 修改前内容
- after: 修改后内容
- reason: 修改理由（引用具体失败案例 ID 和 Rubric 维度）
- trigger_case_id: 触发此提案的主要失败案例 ID
- risk_score: 0.0-1.0 风险评分（高风险=可能影响已通过案例）
```

**Proposer 模型选型：** 使用强推理能力模型（如 claude-opus-4-8 或 GPT-4o），因为提案生成需要理解失败模式并提出有创意的修复方案。

### 4.2 Critic 模型

Critic 是 Proposer 的守门人，负责在提案进入人工审查前过滤低质量提案。

**Critic 评估维度：**

| 维度 | 检查内容 | 拒绝阈值 |
|------|----------|---------|
| 覆盖率 | before 内容是否实际出现在当前配置中 | before 不存在则拒绝 |
| 针对性 | after 内容是否直接响应 trigger_case 的失败原因 | 相关性 < 0.4 拒绝 |
| 无害性 | after 内容是否包含不当输出/幻觉内容 | 安全分 < 0.8 拒绝 |
| 重复性 | 是否与历史已接受提案实质重复 | 相似度 > 0.85 拒绝 |

```python
class Critic:
    async def filter(self, proposals: list[Proposal], agent: AgentConfig) -> list[Proposal]:
        filtered = []
        for proposal in proposals:
            scores = await self._evaluate(proposal, agent)
            if all([
                scores.coverage >= 0.5,
                scores.relevance >= 0.4,
                scores.safety >= 0.8,
                scores.novelty >= 0.15,
            ]):
                proposal.risk_flag = scores.risk_score > 0.6
                filtered.append(proposal)
        return filtered
    
    async def _evaluate(self, proposal: Proposal, agent: AgentConfig) -> CriticScores:
        # 使用更快/更便宜的模型（如 claude-haiku-4-5）
        response = await llm.complete(
            model="claude-haiku-4-5-20251001",
            prompt=CRITIC_PROMPT.format(proposal=proposal, agent=agent)
        )
        return CriticScores(**json.loads(response))
```

### 4.3 历史感知（避免重复）

每次 Proposer 运行时，注入最近 N 次被拒绝的提案摘要：

## 5. 人工审查闸门与回归门控

### 5.1 人工审查闸门（UI 层）

这部分逻辑已在前端 Demo 中完整模拟：

- **Pending** → 用户点击 ✓ → **Accepted**
- **Pending** → 用户编辑后确认 → **Edited**（`editedContent` 存储最终内容）
- **Pending** → 用户点击 ✗ → **Rejected**

后端在用户提交审查后，将所有决策写入 `proposals` 表的 `status`、`edited_content`、`reviewed_by`、`reviewed_at` 字段。

### 5.2 回归门控算法

回归门控是防止过拟合的核心安全机制。只有通过门控的提案才能真正合并到 Agent 配置。

**门控逻辑（伪代码）：**

```python
class RegressionGate:
    # 训练集提升阈值：至少提升 2%
    TRAIN_IMPROVEMENT_MIN = 0.02
    # 保留集降级容忍度：最多下降 1%
    HOLDOUT_DEGRADATION_MAX = -0.01
    
    async def evaluate(
        self, 
        agent_id: str,
        accepted_proposals: list[Proposal],
        baseline: BaselineScores
    ) -> GateResult:
        
        # 构建临时 Agent 配置（应用已接受提案）
        whitebox = db.get_agent_whitebox(agent_id)
        candidate_whitebox = apply_proposals(whitebox, accepted_proposals)
        
        # 对 train 和 holdout 集合重新运行评估
        dataset = db.get_dataset(agent_id)
        
        train_results = await run_agent_batch(candidate_whitebox, dataset.train)
        holdout_results = await run_agent_batch(candidate_whitebox, dataset.holdout)
        
        train_score_after = await judge.score_batch(train_results, candidate_whitebox.rubric)
        holdout_score_after = await judge.score_batch(holdout_results, candidate_whitebox.rubric)
        
        train_delta = train_score_after - baseline.train
        holdout_delta = holdout_score_after - baseline.holdout
        
        # 门控判断
        train_passed = train_delta >= self.TRAIN_IMPROVEMENT_MIN
        holdout_passed = holdout_delta >= self.HOLDOUT_DEGRADATION_MAX
        gate_passed = train_passed and holdout_passed
        
        # 若门控失败，识别导致回归的高风险提案并自动拒绝
        auto_rejected_ids = []
        if not gate_passed:
            auto_rejected_ids = await self._identify_regression_culprits(
                whitebox, accepted_proposals, baseline, dataset
            )
        
        return GateResult(
            passed=gate_passed,
            train_delta=train_delta,
            holdout_delta=holdout_delta,
            auto_rejected_ids=auto_rejected_ids
        )
    
    async def _identify_regression_culprits(
        self, 
        baseline_whitebox, 
        proposals, 
        baseline_scores,
        dataset
    ) -> list[str]:
        """逐一 ablation：移除每条提案后重测，找出导致 holdout 下降的元凶"""
        culprits = []
        for p in proposals:
            # 应用除该提案外的所有提案
            partial = apply_proposals(baseline_whitebox, [x for x in proposals if x.id != p.id])
            results = await run_agent_batch(partial, dataset.holdout)
            score = await judge.score_batch(results, partial.rubric)
            # 若移除该提案后 holdout 分数恢复，则认定为元凶
            if score - baseline_scores.holdout >= self.HOLDOUT_DEGRADATION_MAX:
                culprits.append(p.id)
        return culprits
```

**门控结果说明（前端已展示）：**

| 场景 | 训练集 Δ | 保留集 Δ | 结论 |
|------|---------|---------|------|
| 完美提升 | +5% | +2% | ✅ 全部通过 |
| 训练过拟合 | +8% | -3% | ❌ 门控失败，ablation 定位元凶并自动拒绝 |
| 无显著提升 | +0.5% | +0.5% | ❌ 不满足最小提升阈值，不写版本 |

## 6. Judge 校准算法

Judge 是评分模型，负责对 Agent 输出打分。其准确性直接决定训练信号质量。

### 6.1 Judge 架构

```
Agent 输出 + Rubric → Judge Prompt → LLM → 结构化分数 + 理由
```

**Judge Prompt 模板：**

```
你是严格的 AI 评审员。根据 Rubric 标准对下方 Agent 输出评分。

== Rubric ==
{rubric_json}

== TestCase ==
输入：{input}
期望输出：{expected}
实际输出：{actual_output}

请以 JSON 格式输出：
{
  "scores": {
    "<dimension_id>": { "score": 0-10, "reason": "..." },
    ...
  },
  "weighted_total": 0-100,
  "pass": true|false,
  "summary": "..."
}
```

### 6.2 Human-LLM 一致性校准

Judge 模型需要定期与人工判断校准，防止评分漂移。

**校准流程：**

1. **抽样**：每 N 条 LLM 判断中，随机抽取 K 条（约 5-10%）发送给人工评审。
2. **比对**：计算 LLM 判断与人工判断的一致性指标（Cohen's Kappa 或 Spearman 相关系数）。
3. **报警阈值**：Kappa < 0.6 时触发校准告警，暂停该 Judge 的自动判断。

```python
class JudgeCalibrator:
    CALIBRATION_SAMPLE_RATE = 0.08  # 8% 抽样
    MIN_KAPPA_THRESHOLD = 0.6
    
    async def sample_for_human_review(self, run_id: str, judgments: list[JudgeResult]):
        """随机抽取判断结果，创建人工复核任务"""
        sample_size = max(1, int(len(judgments) * self.CALIBRATION_SAMPLE_RATE))
        sampled = random.sample(judgments, sample_size)
        
        for j in sampled:
            db.insert_calibration_task(
                judgment_id=j.id,
                run_id=run_id,
                status='pending'
            )
    
    def compute_kappa(self, llm_judgments: list[str], human_judgments: list[str]) -> float:
        """Cohen's Kappa：衡量 LLM vs 人工判断一致性"""
        from sklearn.metrics import cohen_kappa_score
        return cohen_kappa_score(llm_judgments, human_judgments)
    
    def check_calibration(self, agent_id: str) -> CalibrationStatus:
        recent_tasks = db.get_completed_calibration_tasks(agent_id, limit=100)
        if len(recent_tasks) < 10:
            return CalibrationStatus(valid=True, kappa=None, message="样本不足，暂不校验")
        
        kappa = self.compute_kappa(
            [t.llm_judgment for t in recent_tasks],
            [t.human_judgment for t in recent_tasks]
        )
        
        if kappa < self.MIN_KAPPA_THRESHOLD:
            return CalibrationStatus(
                valid=False, kappa=kappa,
                message=f"Judge 一致性过低（κ={kappa:.2f}），建议重新校准"
            )
        return CalibrationStatus(valid=True, kappa=kappa)
```

### 6.3 Rubric 权重自适应（进阶）

基于历史人工判断数据，可以训练一个简单回归模型，自动优化 Rubric 各维度权重，使 LLM Judge 的加权总分与人工总分最大程度对齐：

```python
def optimize_rubric_weights(agent_id: str) -> dict[str, float]:
    """使用历史校准数据优化 Rubric 权重"""
    calibration_data = db.get_calibration_history(agent_id, limit=200)
    
    # X: LLM 各维度分数矩阵, y: 人工总分
    X = np.array([[d.dimension_scores[dim] for dim in rubric_dims] for d in calibration_data])
    y = np.array([d.human_total_score for d in calibration_data])
    
    # 约束：权重非负且和为 1
    from scipy.optimize import minimize
    def loss(weights):
        predicted = X @ weights
        return np.mean((predicted - y) ** 2)
    
    result = minimize(
        loss, x0=np.ones(len(rubric_dims)) / len(rubric_dims),
        constraints={'type': 'eq', 'fun': lambda w: sum(w) - 1},
        bounds=[(0, 1)] * len(rubric_dims)
    )
    
    return dict(zip(rubric_dims, result.x))
```

## 7. AgentVersion 快照与 Diff 算法

### 7.1 不可变快照设计

每次训练完成后，AgentVersion 作为完整白盒配置的 JSON 快照写入 S3，附带哈希校验：

```python
def create_version_snapshot(agent_id: str, whitebox: AgentWhiteboxConfig) -> str:
    snapshot = {
        "agent_id": agent_id,
        "timestamp": datetime.utcnow().isoformat(),
        "whitebox": whitebox.to_dict(),
        "schema_version": "1.0"
    }
    content = json.dumps(snapshot, ensure_ascii=False, sort_keys=True)
    sha256 = hashlib.sha256(content.encode()).hexdigest()
    
    s3_key = f"agents/{agent_id}/versions/{sha256[:8]}.json"
    s3.put_object(Key=s3_key, Body=content, ContentType="application/json")
    
    return s3_key
```

### 7.2 版本 Diff 算法

用于 "对比两个版本差异" 的界面功能（Timeline 中的版本对比）：

```python
def diff_versions(v1: AgentVersion, v2: AgentVersion) -> VersionDiff:
    s1 = v1.snapshot['whitebox']
    s2 = v2.snapshot['whitebox']
    
    diffs = []
    
    # Instruction Segments diff
    segs1 = {s['id']: s['content'] for s in s1.get('instruction_segs', [])}
    segs2 = {s['id']: s['content'] for s in s2.get('instruction_segs', [])}
    for seg_id in set(segs1) | set(segs2):
        if segs1.get(seg_id) != segs2.get(seg_id):
            diffs.append(DiffEntry(
                unit='instruction', unit_id=seg_id,
                before=segs1.get(seg_id, '[新增]'),
                after=segs2.get(seg_id, '[删除]')
            ))
    
    # Few-shots diff（按 input 内容匹配）
    fs1 = {fs['input'][:50]: fs for fs in s1.get('few_shots', [])}
    fs2 = {fs['input'][:50]: fs for fs in s2.get('few_shots', [])}
    added_fs = set(fs2) - set(fs1)
    removed_fs = set(fs1) - set(fs2)
    for key in added_fs:
        diffs.append(DiffEntry(unit='few-shot', before='', after=json.dumps(fs2[key])))
    for key in removed_fs:
        diffs.append(DiffEntry(unit='few-shot', before=json.dumps(fs1[key]), after=''))
    
    # Retrieval config diff
    rc1 = s1.get('retrieval_config', {})
    rc2 = s2.get('retrieval_config', {})
    if rc1 != rc2:
        diffs.append(DiffEntry(
            unit='retrieval', before=json.dumps(rc1), after=json.dumps(rc2)
        ))
    
    return VersionDiff(from_version=v1.version_tag, to_version=v2.version_tag, entries=diffs)
```

## 8. 程序性记忆写门（Procedural Memory Write Gate）

程序性记忆（Procedural Memory）指 Agent 从运行中学习到的隐式经验，包括"何时需要人工介入"、"哪类问题最易出错"等模式。

### 8.1 写门设计原则

- **置信度阈值**：仅当新观察的置信度 ≥ 0.8 时才写入。
- **去重**：写入前检查与现有记忆的语义相似度，避免冗余。
- **时间衰减**：旧记忆每 30 天降低 0.1 的权重，防止过时知识持续影响。
- **版本归属**：每条记忆记录生成时的 AgentVersion，便于追溯和清理。

### 8.2 实现方案

```python
class ProceduralMemoryWriteGate:
    CONFIDENCE_THRESHOLD = 0.8
    DEDUP_SIMILARITY_THRESHOLD = 0.85
    
    def __init__(self, embedding_model, vector_store):
        self.embed = embedding_model
        self.vs = vector_store  # pgvector
    
    async def attempt_write(
        self, 
        agent_id: str, 
        memory: MemoryEntry,
        source_run_id: str
    ) -> WriteResult:
        
        # 1. 置信度检查
        if memory.confidence < self.CONFIDENCE_THRESHOLD:
            return WriteResult(written=False, reason=f"置信度不足: {memory.confidence:.2f}")
        
        # 2. 语义去重
        embedding = await self.embed(memory.content)
        similar = await self.vs.search(
            table="procedural_memories",
            embedding=embedding,
            filter={"agent_id": agent_id},
            threshold=self.DEDUP_SIMILARITY_THRESHOLD,
            limit=1
        )
        if similar:
            # 更新现有记忆的观测计数
            await self.vs.increment_observation_count(similar[0].id)
            return WriteResult(written=False, reason="与现有记忆重复", merged_into=similar[0].id)
        
        # 3. 写入
        memory_id = await self.vs.insert(
            table="procedural_memories",
            content=memory.content,
            embedding=embedding,
            metadata={
                "agent_id": agent_id,
                "source_run_id": source_run_id,
                "confidence": memory.confidence,
                "category": memory.category,
                "created_at": datetime.utcnow().isoformat(),
                "weight": 1.0
            }
        )
        return WriteResult(written=True, memory_id=memory_id)
    
    async def apply_time_decay(self, agent_id: str):
        """定时任务：对旧记忆施加时间衰减"""
        cutoff = datetime.utcnow() - timedelta(days=30)
        await db.execute("""
            UPDATE procedural_memories
            SET weight = GREATEST(0.1, weight - 0.1)
            WHERE agent_id = %s AND created_at < %s AND weight > 0.1
        """, agent_id, cutoff)
```

```sql
  WITH (lists = 100);
```

### 9.2 检索逻辑

```python
class AgentRetriever:
    async def retrieve(
        self, 
        query: str, 
        agent_id: str,
        config: RetrievalConfig
    ) -> list[RetrievedChunk]:
        
        # 生成查询向量
        query_embedding = await embed(query)
        
        # 构建过滤条件
        tag_filter = ""
        if config.tag_filters:
            tag_filter = f"AND tags && ARRAY[{','.join(repr(t) for t in config.tag_filters)}]"
        
        # 向量相似度搜索 + Agent 关联过滤
        results = await db.query(f"""
            SELECT doc_id, chunk_index, content,
                   1 - (embedding <=> %s::vector) AS similarity
            FROM knowledge_embeddings
            WHERE %s = ANY(linked_agents)
            {tag_filter}
            ORDER BY embedding <=> %s::vector
            LIMIT %s
        """, query_embedding, agent_id, query_embedding, config.top_k)
        
        return [RetrievedChunk(
            doc_id=r.doc_id,
            content=r.content,
            similarity=r.similarity
        ) for r in results]
```

### 9.3 RetrievalConfig 热更新

当 Proposal 中包含 `retrieval` 类型的提案被接受后，新的 `topK` 和 `tagFilters` 立即生效，无需重新嵌入文档：

```python
def apply_retrieval_proposal(agent_id: str, proposal: Proposal):
```

## 10. 模型路由设计

平台中存在多个 LLM 调用点，不同任务对模型能力和延迟的要求不同：

| 调用场景 | 推荐模型 | 理由 |
|--------|---------|------|
| Agent 实际推理（运行测试用例） | claude-sonnet-4-6 | 平衡能力与速度 |
| Proposer（生成训练提案） | claude-opus-4-8 | 需要深度分析和创造力 |
| Critic（过滤提案） | claude-haiku-4-5-20251001 | 高吞吐量，每条提案都需调用 |
| Judge（评分） | claude-sonnet-4-6 | 准确性优先，可并发 |
| 文档嵌入（Embedding） | text-embedding-3-small | 成本低，质量满足需求 |

### 10.1 路由实现

```python
class ModelRouter:
    ROUTE_MAP = {
        "agent_inference": "claude-sonnet-4-6",
        "proposer":        "claude-opus-4-8",
        "critic":          "claude-haiku-4-5-20251001",
        "judge":           "claude-sonnet-4-6",
    }
    
    def get_model(self, task: str, override: str | None = None) -> str:
        if override:
            return override
        return self.ROUTE_MAP.get(task, "claude-sonnet-4-6")
    
    async def complete(self, task: str, prompt: str, **kwargs) -> str:
        model = self.get_model(task, kwargs.pop('model_override', None))
        client = anthropic.AsyncAnthropic()
        message = await client.messages.create(
            model=model,
            max_tokens=kwargs.get('max_tokens', 4096),
            messages=[{"role": "user", "content": prompt}]
        )
        return message.content[0].text
```

### 10.2 成本控制

- **批处理**：Judge 对同一测试集的评分调用合并为批次，每批最多 20 条，使用 `asyncio.gather` 并发执行。
- **缓存**：相同 (input, agent_version) 对的 Agent 推理结果缓存 24 小时（Redis），避免重复计算。
- **Critic 预过滤**：Proposer 生成的提案先经 Critic 轻量过滤，减少进入昂贵 Judge 评估的提案数量。
