## 1. 一句话定义

一个面向**非技术业务用户**的 Agent 训练工作台:一个用户下可管理多个专精不同任务的 Agent,平台不只让用户"配置" Agent,更让用户像批改作业一样,通过提供案例和判断对错,把 Agent 在某类任务上**越练越精、且不退步**。

---

## 2. 背景与问题

现有同类产品(含内部 demo)已经能搭出 Agent 的"资产层"——统一知识库、统一规则库、统一工具库、评估追踪、接口中心。但它们普遍停在两个状态:

1. **配置态**:用户能调 prompt、参数、绑知识库,但调完不知道变好还是变坏,改进靠玄学。
2. **成绩单态**:评估模块只展示"考完试的分数"(通过率、质量分),没有从失败回流到配置的闭环。

真正的缺口是**训练闭环**:让评估的失败结果自动反推出可执行的改进,再由人审核落地。这恰好是本平台的核心差异点。

另一个被低估的约束:目标用户是不懂代码、不懂 AI 原理的业务员工(合规专员、运营、客服主管等)。他们**不会写 prompt,但极擅长判断"这个回答对不对"**。产品必须把工程复杂度全部藏到背后,只暴露"判断"这一个动作。

---

## 3. 产品目标与设计原则

**核心理念:用户当裁判,LLM 当工程师。** 用户只提供"什么是好的标准"和"这条对不对",所有 prompt、规则、示范、检索配置的改动都由系统在背后生成,用户从头到尾不需要看到"prompt"这个词。

四条设计原则贯穿全文:

- **P1 评估驱动**:评估不是事后考试,而是产品的中心动作;Agent 的改进是评估循环掉出来的副产品。
- **P2 全程白盒**:Agent 的可训练状态拆成一组具名、可读、可单独编辑的单元,绝不优化成一坨不可读的 prompt。用户随时能看到"改了什么、为什么改",随时能接管。
- **P3 声明式配置**:用户在工作台写的一切都是声明式 config,不直接执行;运行时把它"编译"成一次具体的上下文与执行图。由此天然得到版本化、diff、回滚、A/B、草稿/发布分离。
- **P4 训练 = 配置优化,不是调权重**:机制上是"评估失败 → 反推 config 改动 → 守住回归"的闭环。对用户话术可叫"训练",但底层是可解释、即时生效、可回滚的配置演进。

## 4. 用户与角色

| 角色 | 典型身份 | 在平台里做什么 |
|---|---|---|
| 业务训练者 | 合规专员、运营、客服主管 | 创建 Agent、提供案例、当裁判判对错、审查系统提的改动、发布 |
| 资产维护者 | 业务骨干 / 小管理员 | 维护统一知识库、规则库、工具库等共享资产 |
| 平台管理员 | IT / 平台方 | 配额治理、权限、模型路由、接口与鉴权 |
| 接入方(外部) | 下游业务系统 | 通过接口中心调用已发布 Agent |

v1 主角是**业务训练者**;其余角色围绕其训练闭环服务。

## 5. 核心概念与领域模型(概念层)

实体及关系(概念层面,非库表设计):

| 实体 | 说明 | 关键关系 |
|---|---|---|
| User | 平台用户 | 拥有多个 Agent 与全部共享资产 |
| Agent | 专精某类任务的智能体,有用途、状态、当前版本指针 | 属于 User;指向一个当前 AgentVersion |
| **AgentVersion** | Agent 可训练状态的**不可变快照**,带质量分 | 由"训练轮"或"人工编辑"产生,可 diff、可回滚 |
| 可训练状态 | 指令分段 + 规则绑定 + Few-shot 示范 + 检索配置 + 参数 | 是 AgentVersion 的内容,每项均为白盒单元 |
| KnowledgeCollection / Document / Chunk | 用户级统一知识库 | Agent ⇄ Collection 为 M:N,**绑定关系上挂 per-agent 检索配置** |
| Rule | 用户级统一规则库的卡片(名称/说明/优先级/分类/标签) | Rule ⇄ Agent 为 M:N |
| Tool | 用户级统一工具库(能力说明/分类/endpoint/统一 I/O schema/when_to_use) | Tool ⇄ Agent 为 M:N |
| **Rubric / Criterion** | Agent 的评分标准,拆成多个维度(维度/权重/打分指引) | 一等实体,被 Judge 复用、被版本追踪、被校准 |
| **EvalCase** | 评估用例(输入/期望/用户判断/来源/切分标记 train\|holdout) | 来源可为:用户纠正、LLM 生成、真实调用回灌 |
| EvalRun | 某 AgentVersion 在某评估集上的一次跑分 | 产出通过率、质量分、失败用例 |
| **Proposal** | 一轮训练中,系统对某个白盒单元提的一条改动 diff | 带诊断理由、触发用例链接、审查状态 |
| TrainingRound | 一次训练:失败 → 诊断 → 提案 → 审查 → 守门 → 升版 | 串联 EvalRun / Proposal / 新 AgentVersion |
| Run / Trace | 每次执行的完整轨迹(step 级) | 调试与可观测的唯一抓手 |
| Memory | 训练沉淀的 Procedural Memory(学到的规则 + 示范) | scope:user\|agent,可跨 Agent 复用 |

设计要点:**知识/规则/工具一律"引用"而非"拷贝"**。改一处,所有引用它的 Agent 同步生效;真正 per-agent 的不是资产本身,而是绑定关系上的检索/启停配置。

## 6. 功能模块

### 6.1 矩阵看板

以二维矩阵形式总览全部业务节点的 Agent 覆盖情况。行为业务域（如合规、结算、COP、门店运营等），列为处理阶段（如审查、分诊、生成、复核等），单元格显示该节点是否已有 Agent 落位及其发布状态。

- **状态筛选**：按已发布 / 训练中 / 规划中过滤，快速定位空白节点。
- **热力图模式**：用颜色密度展示各业务域的 Agent 覆盖程度，识别薄弱区域。
- **单元格下钻**：点击单元格展开该节点的 Agent 详情、状态和质量分。
- **Agent 落位**：将已有 Agent 指定到某个业务节点，形成业务节点 → Agent 的映射关系。

矩阵看板是平台的"全局视图"入口，帮助业务管理者快速判断哪些节点有 Agent、哪些还在空白。

### 6.2 Agent 管理

Agent 列表与生命周期管理。每个 Agent 有用途描述、类型、业务节点归类、当前质量分、状态（草稿 / 待训练 / 已训练 / 已发布）。支持新建、复制（继承资产绑定与当前版本作为新起点）、启停、删除。Agent 详情页是训练闭环的入口。

**业务蓝图（Blueprint）**：新建 Agent 时，系统根据用户填写的名称、类型和用途，自动匹配一个预设业务蓝图（如合同审查 / 工单分诊 / 披露合规），预填工作流步骤、工具配置、测试用例样本和 guardrails，大幅降低冷启动成本。用户可在详情页自由调整。

**上线预检分数（Readiness Score）**：Agent 详情页顶部展示综合预检结论，由知识库覆盖率、规则覆盖率、工具可用性、测试通过率和当前质量分加权合成。告诉用户距离可发布还缺什么——是该补知识、补规则，还是补测试样例。

### 6.3 统一知识库

用户级文档库，支持上传、分类、打标签、片段预览、增删。**分类用 metadata tag + filter 而非刚性目录**，支持同一文档归多类。提供"检索测试"输入框：输入查询即时看命中片段，是用户调检索效果的反馈面。外部源（如 Notion / 云盘）走增量 sync 而非一次性导入。

### 6.4 统一规则库

业务规则资产化为卡片（名称、说明、优先级、分类、标签），并显示被哪些 Agent 引用。Agent 详情页只做"选择 + 启停"，规则正文在规则库统一维护。规则是训练闭环可写入的目标之一（系统可提"新增/修改某条规则"的提案）。

**规则隔离类型**：Agent 详情页中的规则分为两类：
- **共享规则（binding）**：直接引用规则库已有卡片，修改后全部引用该规则的 Agent 同步生效。
- **草稿规则**：per-agent 的私有草稿，分两种用途——**新建草稿（create）** 指尚未入库的新规则，**内容变更申请（content）** 指对共享规则的本地修改请求，需经审核才能回写规则库。这一机制保证共享资产的稳定性，同时允许 Agent 先行试验规则变更。

### 6.5 统一工具库

把检索、规则校验、生成、路由、人工复核、接口网关等沉淀为可复用工具。**所有工具统一到一套 I/O schema**（输入契约、输出契约、`when_to_use` 描述），内置工具、HTTP/API、MCP server、甚至 sub-agent 都套同一壳。这直接解决"工具生态没有统一 I/O schema"的历史痛点，也是后续编排的基础。

### 6.6 Skill 库

Skill 是比单个工具更高一层的可复用单元：**一个命名、版本化的 SOP 步骤包**，包含一组有序步骤，每步带输出锚点定义（见 6.8）。Skill 库集中管理这些步骤包，供 Agent 工作流引用。

- **Skill 属性**：名称、版本号、分类（检索 / 校验 / 生成 / 路由 / 人工复核）、说明、步骤列表、被引用 Agent 数。
- **步骤锚点**：每个步骤内嵌输出锚点，定义该步骤必须产出哪些结构化字段。
- **引用方式**：Agent 工作流的某个步骤可引用一个 Skill，相当于把该 Skill 的 SOP 作为该步骤的标准执行路径。Skill 更新后，引用它的所有 Agent 步骤可获得新版本。
- **生命周期**：Skill 独立创建、版本化管理，与 Agent 解耦——修改 Skill 不直接改动 Agent，Agent 需主动升级引用版本。

### 6.7 Agent 可训练状态（白盒配置单元）—— 反黑箱的根

这是 P2 的落点。Agent 的"大脑"不是一坨 prompt，而是以下一组具名单元，每一项人都能直接读、直接改：

- **I/O Schema**：声明该 Agent 的输入字段列表和输出字段列表，作为编排层和接口层的契约。
- **指令分段**：把指令拆成角色 / 任务 / 约束 / 输出格式等段落，改哪段 diff 哪段，绝不整段重写。
- **规则绑定**：引用规则库中的卡片，可启停，并可标注隔离类型（见 6.4）。
- **Few-shot 示范**：一组"输入 → 期望输出"的例子。这是非技术用户唯一天然会的训练方式——举例子教，也是最白盒的杠杆。
- **检索配置**：绑哪些知识库、top_k、标签过滤等。
- **参数**：temperature 等，有界，且优先由系统自动调而非让用户面对原始数值。

关键设计：**"自动训练"和"人工编辑"共用同一个编辑面**。系统训练 = 往这堆单元上提 diff；人工调整 = 人往同一堆单元上提 diff。两者走同一套审查 → 版本 → 评估管道。人工编辑因此不是旁路，而是训练流程里的审查闸门。

### 6.8 工作流管理（SOP + 输出锚点 + 路由规则）

Agent 详情页的工作流编辑器，让用户声明式地定义该 Agent 的执行路径，是 6.7 白盒配置单元的结构化延伸。

**工作流步骤（SOP）**：有序的步骤列表，每步有标题、说明和启停开关。步骤可引用 Skill 库中的 Skill，复用标准化执行路径。

**步骤输入字段声明**：每个步骤显式声明它读取哪些上游锚点字段（通过字段名匹配）。声明后，执行引擎在运行时自动从上游步骤的锚点输出中提取对应字段传入，无需人工指定搬运路径。在 trace 中可精确查看每个步骤从哪里读到了哪个值，是步骤级追溯的基础。

**输出锚点（Output Anchor）**：每个步骤可定义一组结构化输出字段，作为该步骤的"完成标准"：
- 字段类型：枚举（enum）/ 布尔（bool）/ 文本（text）/ 数值（number）/ 列表（list）
- 必填标记：声明哪些字段是该步骤必须输出的
- **跨字段约束（点选 DSL）**：以点选方式声明字段间的依赖条件，形式为 "如果 [字段A] [运算符] [值] → 则 [字段B] 必须为 [值]"。运算符支持：等于 / 不等于 / 大于 / 大于等于 / 小于 / 小于等于。例："如果 risk_level = 高 → 则 requires_human_review 必须为 true"。约束以结构化规则存储，执行引擎可机器验证，违反时在 trace 中精确定位到步骤和字段，而非事后猜测。
- strict 模式：是否要求严格符合锚点定义，不允许缺字段

**路由规则（Routing）**：每个步骤可配置条件跳转，替代默认的顺序执行：
- **条件以结构化三元组声明**：[字段] [运算符] [值]，运算符支持等于 / 不等于 / 大于 / 小于，值从字段类型的合法值中选取。例："conclusion = 拒绝 → 跳转到 人工复核"。条件同样以结构化规则存储，可在 trace 中追踪哪条路由被触发。
- 条件满足 → 跳转到指定步骤；条件满足 → 终止流程；无路由规则时步骤按序执行

工作流配置本身是声明式的，运行时由执行引擎将其编译为实际调用图。所有约束和路由条件均作为结构化规则存储——不是提示词里的一句话，而是执行引擎可验证、trace 可精确引用的规则条目。

### 6.9 评估体系（Rubric / 评估集 / Judge / 校准）

**Rubric 是一等实体。** 不是一句"要专业准确"，而是结构化的多维度评分标准（维度 + 权重 + 每维打分指引）。系统提供 **LLM 辅助立标准**：给用户看几条输出，问"这条 ok 吗、为什么"，从用户的零散反馈里反推出显式 rubric。

**冷启动用生成而非让用户写。** 用户手上没有标好的案例，系统按任务先生成候选用例，用户只做 👍/👎 + 一句原因。每次判断同时成为训练信号和回归用例。

**Judge 校准（强制前置）。** 在用 Judge 打分前，必须有一步：让用户标 N 条、LLM 也标、算一致率，不一致就回去改 rubric，直到一致率达标才允许 Judge 自动打分。**没有这一步，所有分数都是在噪声上优化出来的假分。**

**Judge 校准阶段**：随着样例数增长，Judge 的自动化程度分三阶段推进：人工标注（前 20 条）→ 平行验证（20-50 条，人工与 LLM 同时标，对比一致性）→ 自动裁判（50 条以上，LLM 主导打分，人工抽检）。阶段由平台根据用例数自动推进，用户可查看当前所处阶段。

### 6.10 训练闭环（核心）—— 六步

一轮训练的完整链路。括号标注哪些对业务用户可见：

1. **运行评估（⚙️内部 / 结果可见）**：在评估集上跑当前版本。评估集按 train / holdout 切分，holdout 不参与优化，只用于防过拟合。产出质量分、通过率、失败用例。
2. **失败聚类 + 诊断（⚙️内部）**：把失败聚成"主题"，再由 critic 模型给自然语言根因。用户看到的是模式（如"还不会核对授权口径与披露口径是否一致"），不是一堆个例。
3. **生成改动 / 提 diff（⚙️内部）**：proposer 模型把诊断落成针对具名单元的 diff（Few-shot / 规则 / 检索配置 / 指令分段 / 参数），每条 diff 带诊断理由与触发用例链接。
4. **人工审查闸门（🟢用户操作）**：整个闭环里业务用户**唯一**要动手的地方。每条改动以卡片呈现，用户逐条**接受 / 编辑 / 拒绝**。改动卡片额外展示：风险标记（⚠ 过拟合风险）、依赖关系（该改动依赖另一条才生效）、冲突检测（与已有配置冲突时提示）。这是反黑箱的核心——用户随时看到改什么、为什么改，随时接管。
5. **回归守门（⚙️内部）**：把通过审查的改动应用成候选版本，在 train 与 holdout 上重跑。规则：**训练集要涨，且留出集不能掉**。任何只让训练集涨、却拖累留出集的改动，自动剔除（典型过拟合拦截）。
6. **升版提交（⚙️ / 🟢结果可见）**：守门通过则提交新版本（v0.1 → v0.2，质量分上升，状态 待训练 → 已训练）。沉淀的 Few-shot + 规则写入 Procedural Memory，可被其它 Agent 复用；原失败用例进入回归集，日后退化立即被抓出。升版提交完成后，系统自动生成本轮**训练摘要卡（Lesson Summary）**，摘要卡在训练控制台显眼位置展示，作为下一轮训练的上下文线索。

**训练摘要卡（Lesson Summary）**：每轮训练结束后自动生成。内容面向业务用户，包含三部分：
- **修复成功**：此次训练消除了哪些失败模式（自然语言摘要，用户可读）
- **拦截记录**：有哪些改动尝试被回归守门拦截、未能写入，以及拦截原因（帮助用户理解哪些问题训练尚未解决）
- **建议聚焦**：基于当前剩余失败模式，下一轮训练建议优先处理的方向

摘要卡在下一次启动训练前始终可见。下一轮训练启动时，摘要内容同时作为上下文注入训练引擎，使 proposer 模型在生成新一轮 diff 时能感知上一轮哪些尝试失败过，避免重复提出已被证伪的改动。

**train / holdout 切分**：系统提供默认自动切分，用户也可在测试用例库中手动标记每条用例的归属（训练集 / 留出集），适用于需要把特定边界场景保留为留出集的情况。

**训练的杠杆按"白盒程度 + 安全度"排序**，v1 从上往下做：Few-shot 示范 → 规则 → 检索配置 → 指令分段 → 参数。

### 6.11 评估追踪与可观测

每次执行留 step 级 trace（每步、每个 tool call、token、成本、延迟），是用户 debug 与迭代的唯一抓手。评估追踪页展示：当前 Agent 的测试用例数、最新通过率、质量分、失败用例、版本号、评估运行记录（历史趋势）。

### 6.12 接口中心与发布

已发布 Agent 暴露为带鉴权的 API，提供调用调试器、调用记录、Key 管理。草稿 / 已训练 / 已发布的版本分离，发布即把某个 AgentVersion 冻结为对外服务版本。Agent 详情页内也内嵌发布面板，展示当前版本的 guardrails、接口地址和近期调用记录，方便在训练流程中直接确认并发布。

## 7. 技术架构

### 7.1 总体:Control Plane + Data Plane

- **Control Plane(配置工作台)**:用户声明式地配 Agent——绑资产、立 rubric、攒用例、审查训练提案。产出是声明式 config(AgentVersion)。
- **Data Plane(运行时)**:把某个 AgentVersion**编译**成一次具体的上下文 + 执行图并执行。

把 config 当成"可编译产物"是整个架构的地基:版本化、diff、回滚、A/B、草稿/发布全部由此自然得到。

### 7.2 核心服务(概念组件)

- **Agent Runtime / 统一执行器**:负责上下文组装(指令 + 记忆 + 检索结果 + 工具描述,带预算与优先级)、工具调用、Agent Loop。统一执行器对接 6.5 的统一工具 schema,sub-agent 可直接当工具挂上。
- **检索服务**:向量 + 混合检索 + rerank,按 per-agent 绑定配置(top_k / 标签过滤)执行;支撑知识库的"检索测试"。
- **评估 / Judge 服务**:LLM-as-judge,按 rubric 多维打分;内置**校准流程**(judge 与人工一致率),未达标不允许自动打分。
- **训练引擎(本平台核心)**:封装六步闭环——失败聚类、critic 诊断、proposer 生成 diff、train/holdout 切分、回归守门。critic + proposer 对应 reflective prompt evolution / 文本梯度(TextGrad)思路;Few-shot 优化对应 DSPy 的 BootstrapFewShot / MIPRO;指令重写对应 APE / OPRO。可作为学术对照与实现参考。
- **版本与 diff 存储**:每轮训练或人工编辑产生新的不可变 AgentVersion,支持 diff 展示与一键回滚。
- **Trace / 可观测**:全链路 step 级追踪(Langfuse / Phoenix 模型)。
- **记忆服务**:训练沉淀写入 Procedural Memory,带 **write gate**(置信度打分 + 去重),防止记忆膨胀成噪声;scope 分 user / agent。

### 7.3 两条关键数据流

- **编辑 → 编译 → 执行**:用户改任一白盒单元 → 形成候选 AgentVersion → 运行时编译为上下文 → 执行并留 trace。
- **评估 → 训练 → 升版**:跑评估 → 失败聚类诊断 → proposer 提 diff → 人工审查闸门 → 回归守门 → 提交新版本 → 沉淀记忆 + 回灌回归集。

### 7.4 模型路由

temperature 只是冰山一角,真正影响专精与成本的是**模型路由**:分类/抽取类任务用便宜模型,推理类用强模型;critic / proposer / judge 各自可选不同模型。路由 per-agent 甚至 per-task 可配。

## 8. 详细使用案例(端到端)

**场景**:合规专员小李,不懂代码、不懂 AI,负责审查门店现金流合约。她要训练一个"合同审查 Agent",让它在审合约时越来越准。

**第 1 步 · 创建并描述任务。** 小李在 Agent 管理点"新建",用大白话写用途:"审查门店现金流合约里的收益分配、披露义务、提前终止和争议处理条款。"她绑上知识库里的《合同审查标准》《披露与合规检查清单》,从规则库勾选两条已有规则。她没看到任何 prompt 输入框。

**第 2 步 · LLM 辅助立标准(Rubric)。** 系统让她看了几段示例审查输出,逐条问:"这条算合格吗?为什么?"。小李答:"第二条不行,它没指出收益分配口径缺失。" 系统据此反推出一份结构化 rubric:风险识别完整性(40%)、口径/条款一致性(30%)、修订建议可执行性(20%)、表述清晰(10%),每维带打分指引。她确认。

**第 3 步 · 冷启动攒用例 + 校准 Judge。** 小李没有标好的案例。系统按任务生成 12 条候选合约片段,她逐条👍/👎并补一句原因,几分钟搞定。系统自动把 12 条切成训练集 9 条、留出集 3 条(她不需要理解这一步)。随后系统进入**校准**:它和小李分别给同一批打分,一致率偏低,系统提示 rubric 里"口径一致性"定义太模糊,小李补了一句具体说明,再校准,一致率达标。Judge 自此可信。

**第 4 步 · 首次评估(基线 72)。** 系统在回归集上跑当前 v0.1,质量分 72,通过 9 / 失败 3。失败集中在一类,代表用例是:合约里门店授权"按日采集流水",但披露材料用"周度汇总口径",**Agent 没识别出这处口径不一致**,本该提示修订却放过了。

**第 5 步 · 一轮训练,系统提改动。** 小李点"训练"。系统把 3 条失败聚成一簇"数据授权范围 ↔ 披露口径 · 一致性未校验",critic 给出根因:指令与规则均未覆盖此比对,且《披露与合规检查清单》本含该规则但检索没命中。proposer 提出 4 条改动 diff:
- ① Few-shot 示范 +1:用这条失败合约当例子,教它"日采集 vs 周披露"该判高风险并建议统一口径;
- ② 规则 +1:"数据授权范围与披露口径需一致",优先级高、分类合规;
- ③ 检索配置改:标签过滤加"数据授权、披露口径",top_k 4→6,让合规清单能命中;
- ④(系统预判过拟合,默认不推荐)把该合约原文写死进指令。

**第 6 步 · 人工审查闸门(小李唯一动手处)。** 每条改动是一张卡片,带"为什么改 + 哪条用例触发"。小李接受①;觉得②的措辞不够严,点"编辑"把说明改得更具体后接受;③直接接受;④系统已标红"⚠ 过拟合风险、不推荐",她直接拒绝。全程她看到的都是人话,没碰过 prompt。

**第 7 步 · 回归守门。** 系统把①②③应用成候选 v0.2,在 train 和 holdout 上重跑:训练集 72→88(↑),留出集 70→85(未降),通过守门。(假如小李一时手滑接受了④,守门会发现它在留出集 −9,**自动把④剔除**再提交,并告知原因。)

**第 8 步 · 升版与沉淀。** 提交为 v0.2,质量分 72→88,状态变"已训练"。学到的 Few-shot 示范和规则写入 Procedural Memory,可被其它合规类 Agent 复用;那条原本失败的用例进入回归集——以后只要它退化,下一次评估立即抓出来。小李确认效果后,在接口中心把 v0.2 发布,下游系统即可调用。

**结果**:小李没写一行 prompt、没碰任何 AI 参数,仅靠"提供案例 + 判断对错 + 审查改动",就把 Agent 从 72 练到 88,且明确知道它学到了什么、不会偷偷退步。这就是"用户当裁判、LLM 当工程师"的训练闭环。

## 9. 关键风险与对策

| 风险 | 后果 | 对策 |
|---|---|---|
| Judge 未校准 | 在错误标准上"越练越偏",分数虚假,信任崩塌 | 校准前置为硬门槛：一致率达标才允许自动打分（6.9） |
| 过拟合到少量用例 | 分数虚高,泛化变差 | train/holdout 切分 + 回归守门（6.10 第 5 步） |
| 训练有天花板 | 用户误以为可无限变好 | 明示:config 级训练到顶后需换模型 / 改任务分解 / 补数据,而非继续调 prompt |
| 黑箱化 | 用户不敢用、出事无法追责 | 全程白盒(P2)+ 人工审查闸门 + step 级 trace |
| 强主观任务 rubric 不稳 | 标准立不住,闭环跑不动 | 产品识别此类任务并劝退,优先支持分类/抽取/校验等有明确对错的任务 |
| 资产权限与多租户 | to B 合规与隔离问题 | 资产 scope(user/team/org)与权限模型按需开启;FinTech 场景叠加 guardrails(PII、审核) |

## 10. 路线图(建议)

| 阶段 | 目标 | 关键交付 |
|---|---|---|
| v1(先打通闭环) | 单 Agent 练到专精 | 白盒可训练单元 + Rubric/评估集 + Judge 校准 + **Few-shot 训练闭环**(六步)+ 回归守门 + 版本化 |
| v1.1 | 扩训练杠杆与可观测 | 规则/检索/指令分段提案、失败聚类视图、step 级 trace |
| v2 | 自动化与编排 | cron/webhook/事件触发器、多 Agent 编排(Agent-as-tool / router)、Procedural Memory 跨 Agent 复用 |
| v2+ | 治理与企业化 | 成本/配额治理、guardrails、多租户权限、团队协作 |

**v1 第一刀**:Few-shot 训练闭环 + 人工审查闸门。它是非技术用户最先能跑通、也最不像黑箱的入口。

## 11. 非目标(本期不做)

- 不做模型微调 / 权重训练(训练 = 配置优化)。
- 不做跨 Agent 的多 Agent 编排（router / orchestrator 模式留待 v2）；单 Agent 内的工作流步骤管理（见 6.8）已在 v1 支持。
- 不做强主观、无明确对错任务的训练支持。
- 不在本期处理跨团队协作与复杂多租户权限。
